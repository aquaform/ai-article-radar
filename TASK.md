# Job Card: AI Article Radar

Ты — senior full-stack developer/executor в App Factory. Hermes только оркестрирует и мониторит; всю разработку, git/GitHub, Dokploy и диагностику выполняешь ты.

## Цель
Сделать и задеплоить простое веб-приложение **AI Article Radar**, которое мониторит три русскоязычные площадки — **VC.ru**, **Хабр**, **Дзен** — находит актуальные материалы про ИИ/нейросети/AI, рекомендует их пользователю и показывает короткое саммари.

## Публичный URL
Опубликовать как:

`https://ai-article-radar.apps.sielom.ru`

## Основной сценарий пользователя
1. Пользователь открывает страницу.
2. Видит список свежих AI-статей из VC.ru, Habr и Dzen.
3. У каждой статьи есть:
   - площадка/источник;
   - заголовок;
   - ссылка на оригинал;
   - дата/свежесть, если доступна;
   - короткое саммари на русском (1–3 предложения, эвристически из описания/текста, без LLM API);
   - почему рекомендовано / релевантные теги;
   - оценка релевантности.
4. Пользователь может менять статус статьи:
   - `Новое`;
   - `Хочу прочитать`;
   - `Читаю`;
   - `Прочитано`;
   - `Не интересно`;
   - `Архив`.
5. Статусы должны сохраняться после перезагрузки страницы. Для простоты можно хранить статусы в `localStorage`; серверная БД не обязательна.

## Удобства, которые стоит добавить
- Фильтры по источнику и статусу.
- Поиск по заголовку/саммари/тегам.
- Кнопка `Обновить`.
- Показ `последнее обновление` и статуса источников: сколько статей найдено и какие источники не ответили.
- Быстрое скрытие `Не интересно`.
- Экспорт/импорт статусов JSON или хотя бы экспорт списка `Хочу прочитать` в markdown.
- Mobile-friendly интерфейс для Telegram/Android пользователя.

## Источники и реализация сбора
Сделай серверный агрегатор с кэшем (например 15 минут), чтобы браузер не упирался в CORS.

Площадки:
- Habr: ищи рабочий RSS/search/feed endpoint или HTML fallback.
- VC.ru: ищи рабочий RSS/feed/search endpoint или HTML fallback.
- Dzen: ищи рабочий search/feed endpoint или HTML fallback.

Важно:
- Во время реализации реально проверь запросы к источникам.
- Если один источник блокирует/не отвечает, приложение должно продолжать работать с остальными и явно показывать degraded status.
- Не добавляй фейковые/синтетические статьи как будто они реальные. Можно использовать только реальные найденные URL или честно показать, что источник недоступен.
- Не используй платные API/LLM API/секреты для саммари. Саммари делай эвристически из title/description/snippet/content.

## Технические ожидания
Предпочтительно простой Node.js stack:
- Backend: Fastify/Express + TypeScript или понятный JS.
- Frontend: можно Vite/React или простая SPA без тяжёлых зависимостей.
- Tests: хотя бы юнит-тесты для нормализации/фильтрации/статусов + smoke test `/health` и `/api/articles`.
- Dockerfile в корне.
- Сервис слушает `PORT` (default 3000) на `0.0.0.0`.
- `GET /health` возвращает HTTP 200 и тело `ok`.
- `GET /api/articles` возвращает JSON со статьями и diagnostics по источникам.
- Никаких секретов в коде; не читать и не печатать `.env`.
